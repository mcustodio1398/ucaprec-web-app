Rol de trabajo asumido:
Desarrollador senior full-stack / arquitecto React + Vite + TypeScript + Supabase + Netlify, con prioridad en terminar una versión funcional y publicable hoy.

Objetivo actual:
Dejar la aplicación institucional UCAPREC funcionando de forma estable, conectada a Supabase donde ya existen tablas/vistas, sin datos ficticios visibles en producción, con login inicial y lista para publicar en Netlify.

Cambios realizados en esta sesión:

1. Integración y limpieza de datos ficticios
- Dashboard:
  - Los KPI ya no caen a valores mock; si `vw_dashboard_kpis` no devuelve datos, se muestran ceros.
  - La tabla de expedientes recientes ahora consulta `expedientes` desde Supabase.
  - Si no hay expedientes recientes, muestra estado vacío.
  - La actividad reciente y notificaciones quedaron vacías por defecto para no mostrar alertas inventadas.
- Expedientes:
  - El listado principal carga desde la tabla `expedientes`.
  - Si Supabase falla o no hay datos, se muestra vacío limpio.
- Imputados:
  - El listado carga desde `imputados`.
  - La selección de expediente para agregar imputado carga opciones desde `expedientes`.
- Víctimas:
  - El listado carga desde `victimas`.
- Medidas y Alertas:
  - El listado carga desde `medidas_alertas_expediente`.
- Documentos:
  - El listado carga desde `documentos`.
- Usuarios:
  - El listado carga desde `usuarios`.
- Auditoría:
  - El listado carga desde `auditoria`.
- Se agregó una capa de adaptadores tolerantes a nombres de columnas para mapear datos de Supabase hacia la UI sin depender de un único nombre exacto para cada columna.

3. Navegación de expedientes
- El listado de expedientes ahora conserva el ID seleccionado al abrir detalle o edición.

- Si Supabase devuelve 401, el sistema muestra un mensaje de permisos/RLS en vez de simular guardado.

6. Supabase / 401
- Se creó `supabase_policies_ucaprec.sql` con políticas mínimas de prueba para habilitar lectura e inserción controlada durante pruebas.
- El archivo SQL es temporal y debe reemplazarse por políticas seguras basadas en Supabase Auth antes de producción real.
- Se creó `supabase_reset_schema_ucaprec.sql` como script principal para reconstruir Supabase desde cero:
  - Elimina tablas/vistas actuales.
  - Crea tablas alineadas al frontend actual.
  - Crea `vw_dashboard_kpis` y `vw_expedientes_requieren_atencion`.
  - Inserta roles y usuarios temporales de prueba.
  - Crea catálogos mínimos.
  - Habilita RLS y políticas temporales de lectura/escritura para pruebas.
- Antes de ejecutar el reset completo, confirmar que no hay datos reales que deban conservarse.
- Revisar en Supabase Project Settings > API:
  - Project URL debe coincidir con `VITE_SUPABASE_URL`.
  - Publishable key debe coincidir con `VITE_SUPABASE_ANON_KEY`.
  - Si la llave fue rotada, actualizar `.env.local` y variables de Netlify.
